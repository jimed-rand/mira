# mira

This is forked project from the original [nabakdev/S905X-ArchLinuxARM](https://github.com/nabakdev/S905X-ArchLinuxARM/) for making the ALARM (Arch Linux ARM) for FiberHome HG680P and ZTE B860H (the most cheapest STBs you got and popular in some countries). The name of this forked project is inspired by an VTuber named [Mira Fridayanti](https://virtualyoutuber.fandom.com/id/wiki/Mira_Fridayanti) which she made me inspired to bring me back into VTubing (yeah, I am also VTuber) and I am also want to make the implementation into Go language for this project (coming soon, for now I want to improve something here for my own experiment before I implement into Go language).

---

## Features

- **Arch Linux ARM (aarch64)**
- **UEFI support**
- **GPT partition table**
- **XFS filesystem**
- **Flexible swap options**
- **GRUB bootloader**

## Disk Layout

The project uses a modern GPT partition table with full UEFI support (via GRUB). The layout behaves differently depending on whether it's the raw generated image or the deployment layout created by the installer.

### Built Image Layout (Default)
The initial `.img` generated by `build.sh` has the following layout (total size depends on the root filesystem size configured during the build):

| # | Name | Label | Offset | Size | Filesystem | Flags |
|---|---|---|---|---|---|---|
| — | (reserved) | — | 0 MiB | 68 MiB | raw / bootloader | — |
| 1 | `ESP` | `BOOT` | 68 MiB | 256 MiB | FAT32 | `esp` |
| 2 | `ROOT` | `ROOT` | 324 MiB | Variable (Default 1536 MiB) | ext4 | — |

> The first 68 MiB are intentionally left raw to accommodate the vendor bootloader area expected by the Amlogic ROM without conflicting with primary GPT tables.

### Deployment Layout (mira-install)
When deploying the system via the `mira-install` script directly on the STB (to eMMC or SD Card), the script dynamically re-partitions the entire target drive. You are given options for your root filesystem and swap:

| Partition | Name | Label | Size | Filesystem | Notes |
|---|---|---|---|---|---|
| `p1` | `ESP` | `BOOT_TARGET` | 256 MiB | FAT32 | UEFI subsystem and GRUB config |
| `p2` | `SWAP` | `SWAP_TARGET` | 3072 MiB | linux-swap | *Optional* (only created if Swap on target device is chosen) |
| `last` | `ROOT` | `ROOT_TARGET` | 100% (remaining) | ext4 / xfs | Expands to fill the remaining capacity |

### Filesystem Specifics

- **ext4 (Default)**: `mkfs.ext4` is invoked with `-O ^metadata_csum,^64bit` to ensure maximal compatibility with older U-Boot configurations. When installed via `mira-install`, `tune2fs -O fast_commit` is additionally applied.
- **xfs**: Available dynamically as an alternative modern file system during `mira-install`.

## Boot Chain

```
Amlogic ROM
  └─► aml_autoscript   (compiled U-Boot script — triggers USB/MMC probe, reboots into s905_autoscript)
        └─► s905_autoscript  (compiled U-Boot script — loads u-boot.ext into RAM)
              └─► u-boot.ext  (MainLine U-Boot binary, executed at 0x1000000)
                  ├─► EFI/BOOT/bootaa64.efi (GRUB bootloader, modern UEFI approach)
                  │     └─► grub.cfg -> Linux kernel (Image) + DTB
                  └─► extlinux/extlinux.conf (Legacy syslinux boot menu fallback)
```

All `.cmd` source files are compiled at build time with `mkimage` into their binary counterparts. The `.cmd` sources are removed from the final image; only the compiled scripts are shipped.

## Build

### Prerequisites

The build script must run as **root** (it uses `losetup`, `mount`, `dd`, and `mkfs` internally). The following packages are required on the host:

| Tool | Arch Linux | Debian/Ubuntu | Fedora / openSUSE | Void Linux | Slackware |
|---|---|---|---|---|---|
| `parted` | `parted` | `parted` | `parted` | `parted` | `parted` |
| `mkfs.vfat` | `dosfstools` | `dosfstools` | `dosfstools` | `dosfstools` | `dosfstools` |
| `mkfs.ext4` | `e2fsprogs` | `e2fsprogs` | `e2fsprogs` | `e2fsprogs` | `e2fsprogs` |
| `mkfs.btrfs`| `btrfs-progs`| `btrfs-progs` | `btrfs-progs` | `btrfs-progs` | `btrfs-progs` |
| `bsdtar` | `libarchive` | `libarchive-tools`| `libarchive` | `libarchive` | `libarchive` |
| `mkimage` | `uboot-tools` | `u-boot-tools` | `uboot-tools` | `uboot-tools` | `uboot-tools` |
| `xz` | `xz` | `xz-utils` | `xz` | `xz` | `xz` |
| `losetup` | `util-linux` | `util-linux` | `util-linux` | `util-linux` | `util-linux` |
| `qemu-aarch64-static` | `qemu-user-static` | `qemu-user-static` | `qemu-user-static` | `qemu-user-static` | *(Manual / SBo)* |
| `curl` | `curl` | `curl` | `curl` | `curl` | `curl` |

> *For other distributions (or Universal Linux systems), the dependencies are relatively universal. Standard build tools like `dd`, `mount`, `cp`, and `sed` should be available in generic base packages. Install the tool equivalents using your distribution's package manager.*

### Environment Variables

| Variable | Default | Description |
|---|---|---|
| `WORKING_DIR` | `$SCRIPT_DIR/mira-workspace` | Root directory for all build artefacts |

### Building the Image

Karena pipeline CI/CD sudah tidak digunakan lagi pada repositori ini, _image_ Mira sepenuhnya di-build melalui _host_ lokal Anda. Anda dapat menjalankannya secara interaktif maupun dilakukan secara otomatis (_headless_).

#### 1. Interactive Build (Default)
Secara bawaan, Anda cukup menjalankan skrip dengan hak akses _root_. Skrip akan secara interaktif menanyakan Desktop Environment apa yang ingin dipasang.

```bash
# Clone the repository and navigate into it
git clone https://github.com/jimed-rand/mira
cd mira

# Run the builder (Needs root privileges)
sudo bash build.sh
```

Saat dieksekusi, skrip ini akan secara otomatis:
1. Membuat direktori kerja isolasi `mira-workspace`.
2. Mengunduh pustaka _rootfs tarball_ sistem Arch Linux ARM (hanya jika belum ada dalam _cache_ direktori).
3. Melakukan konfigurasi Desktop yang Anda pilih (seperti XFCE, MATE, LXDE, atau CLI).
4. Melakukan _compile_ dan melabeli partisis GPT/UEFI dan kompresi hasil berupa berkas `.img.xz`.

Hasil dari image yang terkompres akan berada letaknya bersamaan dengan berkas `build.sh` berada.

#### 2. Non-Interactive / Automated Build
Anda juga dapat melewatkan perintah interaktif sehingga skrip berjalan secara otomatis secara fungsional. Ini sangat cocok jika Anda ingin membuat skrip rilis Anda sendiri. Atur *Environment Variables* seperti `DESKTOP_ENV` sesaat sebelum eksekusi skrip:

```bash
# Pilihan instalasi DE yang valid: minimal, xfce, mate, lxde, cosmic
sudo DESKTOP_ENV="xfce" bash build.sh
```

#### 3. Customizing the Build
Jika Anda ingin melangkah lebih jauh, Anda juga dapat menimpa beberapa Environment Variables ini sesuai kebutuhan:
- `OUT_FILENAME`: Mengubah nama dasar dari *image* yang akan diekspor (Default: `Mira-ArchLinuxARM-aarch64_S905X`).
- `WORKING_DIR`: Mengarahkan direktori kerja untuk mengunduh _tarballs_ dan *mount points* internal (Default: `./mira-workspace`).
- `ROOTFS_URL`: Menentukan URL unduhan jika Anda ingin menggunakan basis _rootfs tarball_ versi kustom atau dari _mirror_ lain.

**Contoh kombinasi kustomisasi:**
```bash
sudo OUT_FILENAME="Mira-MATE-Edition" DESKTOP_ENV="mate" bash build.sh
```

### Under the Hood: Build Stages

When `build.sh` is executed, it runs through the following sequence of automated stages:

| Stage | Action | Details |
|---|---|---|
| **`[1/5]`** | Download Base System | Fetches the latest Arch Linux ARM `aarch64` rootfs tarball locally using `curl` if it is not already cached in the workspace. |
| **`[2/5]`** | Image Setup & Partitioning | Allocates an empty image file via `dd`. It then uses `parted` to explicitly write a GPT partition table and formats the partitions (`mkfs.vfat` for ESP and `mkfs.ext4` for ROOT). |
| **`[3/5]`** | Loop Mount | Uses `losetup -P` to attach the raw image as loop devices, and mounts the filesystem layers into the `./mnt` directory inside the workspace. |
| **`[4/5]`** | Extract & Overlay | Boot firmware and kernel overlays are mapped into `mnt/boot`. The official Arch Linux ARM tarball is extracted using `bsdtar`. Custom patch overlays (e.g., config files, `mira-install` scripts) are copied verbatim. |
| **`[ * ]`** | *Chroot Execution (Dynamic)* | `qemu-aarch64-static` is embedded into the image to allow arm64 execution on any host architectures. In the `chroot` environment:<br> - Required packages (`grub`, `efibootmgr`, `btrfs-progs`, `xfsprogs`) are installed via `pacman`.<br> - Desktop Environments (e.g., `xfce`, `mate`) are optionally installed.<br> - **GRUB Bootloader** is installed (`grub-install`) and properly configured for UEFI support. |
| **`[5/5]`** | Compile Boot Scripts | Legacy Amlogic boot scripts (`.cmd`) are compiled into binaries using `mkimage` to ensure compatibility with unmodified vendor firmware. |
| **`[6/6]`** | Cleanup & Compress | Filesystems are unmounted, loopback devices are detached, and the raw image is heavily compressed with `xz -9` to significantly reduce its size.

## First Boot

Flash the `.img.xz` image to a **USB drive** (≥ 4 GiB recommended):

```bash
xz -d Mira-ArchLinuxARM-aarch64_S905X.img.xz
sudo dd if=Mira-ArchLinuxARM-aarch64_S905X.img of=/dev/sdX bs=4M conv=fsync status=progress
sync
```

> Replace `/dev/sdX` with your actual block device. **All existing data on the target device will be destroyed.**

> If you don't want to use `dd`, you can use another tool with GUI which helps you like Balena Etcher, Raspberry Pi Imager, GNOME Disks, Rufus with DD method (if you use Windows and run thsi program under WSL2), etc.

### Default Credentials

| Role | Username | Password |
|---|---|---|
| Superuser | `root` | `root` |

### Post-boot: Initialize Pacman Keyring

The official Arch Linux ARM tarball ships without a populated keyring. Run the following **once** after the first boot:

```bash
pacman-key --init
pacman-key --populate archlinuxarm
```
